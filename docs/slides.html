The Problem

KYC today equates verification
with data collection.

To answer a yes/no question like "is this person 21+?", platforms collect name, DOB, Aadhaar number, address, and photo. Then store it. Then get breached.

Fixed Booleans

UIDAI's Aadhaar VC offers AgeAbove18 — one fixed claim. A lender needs "age 25-55". An insurer needs "age 35-45". No pre-computed claim exists.

SD-JWT: AgeAbove18 = Yes. Need 21+? Sorry.

No Duplicate Detection

Same person applies at 5 NBFCs. No privacy-preserving way to detect it. SD-JWT has no cross-platform linkable token that isn't PII.

5 loans, 5 platforms, zero dedup

Breach Liability

Every platform that stores KYC data becomes a breach target. DPDP Act 2023 imposes penalties. The data shouldn't exist on the verifier's side at all.

DPDP: up to Rs 250 crore per breach
The Solution

Don't collect data.
Verify the math instead.

User's device reads their credential, runs a ZK circuit locally, outputs a proof. The verifier checks one equation. Gets a boolean. Stores nothing.

Tier 1 — Custom Proofs

Any threshold, one credential

Verifier sets the condition as a public input. One circuit handles every age threshold, every bracket, every cutoff.

"age ≥ 21" for Karnataka
"age ≥ 25" for Maharashtra
Tier 2 — Composite Proofs

Multiple conditions, one boolean

Combine age + residence + gender in a single proof. Verifier learns only: "all conditions met" or "not met".

"age ≥ 25 AND from TN"
"age ≤ 25 AND female AND from UP"
Tier 3 — Multi-Source

Cross-credential proofs

Aadhaar VC + ITR + credit bureau → one composite eligibility proof. No single issuer builds this.

"age 25-55 AND income 5L-50L
AND credit ≥ 700"

Why ZKP over SD-JWT?

SD-JWT reveals claim values.
ZKP reveals only the result.

Capability Aadhaar SD-JWT Xylem (Groth16 ZKP)
Custom thresholds (age ≥ 21, 25, 55) Fixed AgeAbove18 only Any threshold as public input
Composite checks (age AND state AND income) Discloses each value Single boolean output
Cross-platform dedup No mechanism Deterministic nullifiers
Verifier cross-linking users Credential ID always visible Unlinkable across verifiers
After verifier breach Claim values + credential ID Meaningless hashes + booleans

SD-JWT is a step forward. ZKP is the next step — selective computation, not just selective disclosure. We consume the SD-JWT as input and output a proof. The VC stays untouched.

Use Cases

Where this applies — today.

Lending & NBFCs

Loan Pre-qualification

"Age 25-55 AND income 5L-50L AND credit ≥ 700" — one proof, zero data. Nullifiers detect duplicate applications across platforms.

Insurance

Bracket Pricing

Range proof reveals "age in 35-45 bracket" without revealing DOB. Composite with "metro resident" for urban health plans.

Government

Subsidy & Scholarships

"Age ≤ 25 AND from UP AND income ≤ 2.5L" — one proof replaces document stacks. Nullifier prevents duplicate claims.

Age-Gated Services

State-Specific Compliance

Drinking age varies by state (21 KA, 25 MH). Gaming, alcohol delivery — platform sets threshold as a public input.

Property & Rental

Tenant Verification

Aadhaar (identity) + ITR (income ≥ 3x rent) + EPFO (employed) — landlord gets one boolean. Zero document copies.

KYC Dedup

Cross-Platform Identity

Same person at 5 banks = same nullifier. Banks share nullifiers (not PII) via registry. One hash, zero personal data.
